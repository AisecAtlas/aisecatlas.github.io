> **이 페이지의 결론**: SOAR는 죽지 않았지만 **진화의 끝**에 가까워졌다. 다음 단계는 정해진 일을 빠르게 하는 자동화가 아니라 **정해지지 않은 일도 안전하게 수행**하는 Agentic 자동화다.

## 자동화의 세 세대

각 세대를 같은 기준으로 비교한다.

| 기준 | 1세대 SOAR | Hyperautomation | Agentic SOC |
|---|---|---|---|
| 워크플로 작성 | 수동 코드 | AI 생성 no-code | 자연어 의도 → 자율 계획 |
| 변화 대응 | 매번 엔지니어 수정 | 동적 조정 가능 | 새 상황 자율 추론 |
| 가드레일 | 정책 외부 | 워크플로 내 검증 | 명시적 가드레일 + 평가 |
| 인간의 역할 | 분류와 모든 결정 | 워크플로 설계와 결정 | 가드레일 정의와 예외 처리 |
| 대표 도구 | XSOAR, Splunk SOAR | Torq, Tines | Charlotte AI, Security Copilot, Torq AI SOC |

> **본질의 차이**: SOAR는 "이 알림이 오면 이렇게 처리하라"는 **명시적 규칙**의 자동화다. Agentic은 "이 목표를 달성하라"는 **의도 기반 자율 행동**이다.

## 1세대 SOAR: 무엇이 한계인가

SOAR는 세 가지에서 막혔다.

**유지보수 비용.** 모든 통합과 playbook이 코드 자산이 된다. **벤더 API 변경, 새 위협, 조직 변화**가 있을 때마다 엔지니어 수정이 필요하다.

**확장성.** 통합 수가 늘면 복잡도가 곱셈으로 늘어난다. n개 도구의 n!가지 상호작용을 관리해야 한다.

**적응력 부재.** 한 번도 본 적 없는 패턴은 자동 분류가 불가능하다. **수동 대응으로 회귀**한다.

대표 도구는 여전히 의미가 있다.

| 솔루션 | 위치 |
|---|---|
| **Palo Alto XSOAR** | 시장 점유율 1위, 풍부한 통합과 사례 |
| **Splunk SOAR** | Splunk SIEM 사용자에게 자연스러운 선택 |
| **Microsoft Sentinel Automation** | M365 환경 통합 |

## Hyperautomation: 무엇이 다른가

Hyperautomation은 SOAR의 **운영 부담을 줄이는** 방향으로 진화했다.

**no-code 워크플로 생성.** 자연어로 의도를 기술하면 로직과 API 호출, 데이터 변환을 자동 생성한다.

**동적 적응.** 워크플로 안에 AI 결정 노드를 두어 상황별로 분기한다.

**중립 통합.** 특정 SIEM/EDR에 종속되지 않아 멀티 벤더 환경에 적합하다.

### 대표 도구

#### Torq

**Hyperautomation을 명시적으로 표방**하는 플랫폼이다.

- **2026년 초 Series D $140M** 유치, 평가액 $1.2B
- 정책 코드 기반 AI 가드레일, 사례 관리, 멀티 벤더 통합
- SOAR의 한계를 명시적으로 비판하며 차별화

#### Tines

**유연성과 사용자 경험**에 강점이 있다.

- **Smart Mode**: 수백만 건의 익명화된 성공 실행 데이터를 학습해 다음 단계를 제안
- **AI Action Creator**: 자연어로 워크플로를 설명하면 즉시 로직을 생성
- 학습 곡선이 완만해 빠른 PoC에 적합

## Agentic SOC: 무엇이 가능해지나

가장 큰 변화는 **하나의 워크플로 안에서 자율 판단**이 가능해진 점이다.

### 새로 가능해진 일

**자율 조사.** 알림에서 출발해 가설을 세우고 **여러 시스템을 교차 조회**한 뒤 결론을 정리한다. playbook에 사전 정의되지 않은 흐름도 추론으로 처리한다.

**다단계 추론.** 한 단계의 결과를 보고 다음 단계를 계획한다. 의심스러우면 추가 데이터를 수집하고, 명확하면 대응을 실행한다.

**자연어 인터페이스.** "지난주에 IAM 위협이 많이 발생한 시점과 그 직전 인증 패턴을 보여줘" 같은 질의에 데이터 기반으로 답한다.

**다중 에이전트 협업.** IAM 전문 에이전트, EDR 전문 에이전트, 클라우드 전문 에이전트가 함께 사건을 처리한다.

### 새로 어려워진 일

같은 자율성이 새 위험을 만든다.

**가드레일 정의.** "무엇을 자율로 허용할 것인가"가 핵심 설계 과제다. 좁게 잡으면 가치가 없고, 넓게 잡으면 위험하다.

**감사 가능성.** 에이전트의 결정 근거를 사후에 설명할 수 있어야 한다. **블랙박스 모델은 컴플라이언스에서 위험**하다.

**환각 위험.** LLM이 잘못된 사실을 생성할 수 있다. 보안 영역에선 치명적이다.

**책임 소재.** 자율 행동의 결과에 대한 책임 분담을 처음부터 정의한다.

## 어느 세대에서 시작할까

조직의 현재 상태에 따라 다르다.

### 자동화 없음 → 1세대 SOAR

playbook 자동화가 처음이면 SOAR가 안전한 시작점이다. **반복 작업 제거가 1차 목표**다.

권장: XSOAR (벤더 풍부), Tines (학습 곡선)

### SOAR 운영 중 → Hyperautomation

기존 SOAR의 운영 부담이 느껴지면 Hyperautomation으로 옮겨가는 흐름이 자연스럽다.

권장: Torq, Tines

### 인력 부족 + 알림 폭주 → Agentic SOC

분석가 채용으로 해결되지 않는 규모라면 Agentic SOC를 우선 검토한다.

권장: 기존 보안 스택과 통합되는 도구. Microsoft 환경은 Security Copilot, CrowdStrike 환경은 Charlotte AI.

## 단계별 도입 체크리스트

### 1단계: 자동화 후보 식별

- [ ] 분석가가 가장 자주 하는 반복 작업 Top 10 추출
- [ ] 그 중 결정 분기가 단순한 작업 선별
- [ ] 자동화 효과(시간/비용) 정량화

### 2단계: 가드레일 정의

- [ ] 자율 허용 행동 목록 작성 (예: 격리, 차단)
- [ ] 승인 필요 행동 목록 작성 (예: 자격 무효화)
- [ ] 절대 자동화 금지 행동 목록 작성

### 3단계: 감사 추적 설계

- [ ] 모든 자동 행동을 SIEM에 기록
- [ ] 에이전트 의사결정 근거를 별도 감사 로그에
- [ ] 분기별 자동화 행동 검토 프로세스 설계

### 4단계: 단계적 확장

- [ ] Triage부터 시작 (자율 대응 없음)
- [ ] 신뢰 측정 후 가역적 행동 자율화
- [ ] 마지막에 일부 비가역적 행동까지 자율화

## 한국 시장 노트

**금융권.** 자율 대응은 금감원 보안성 심의에서 면밀히 검토된다. 도입 전에 감사 추적 설계를 명확히 한다.

**공공 부문.** 망분리 환경에서는 클라우드 SaaS형 SOAR가 어렵다. **온프레미스 옵션이 있는 도구**(XSOAR, Splunk SOAR)가 현실적이다.

**대기업.** 멀티 벤더 환경에서는 중립 SOAR(Torq, Tines)를 자주 선택한다. 단일 벤더 종속을 피하려는 의도다.
